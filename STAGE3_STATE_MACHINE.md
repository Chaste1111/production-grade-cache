# Stage 3 — epoll 非阻塞连接状态机

> HTTP 代理缓存服务器第三阶段架构设计

---

## 为什么需要状态机

```
阻塞版（Stage 2）：
  read(fd) → 没数据 → 线程睡着 → 其他人排队
  一次处理一个连接，一条路走到黑

非阻塞版（Stage 3）：
  read(fd) → 没数据 → 立即返回 EAGAIN → 去处理下一个连接
  多个连接交替推进，谁准备好就处理谁
```

代价：每个连接必须记住"上次处理到哪了"——这就是状态机。

## 多连接并发推进

```
                    同一时刻，多个 Connection 各自推进

  Connection A               Connection B               Connection C
  browser_fd=5               browser_fd=6               browser_fd=8
  target_fd=9                target_fd=10               target_fd=-1
  state=READ_RESPONSE        state=READ_REQUEST         state=WRITE_CLIENT
     │                           │                          │
     │ 正在收百度响应              │ 还在读浏览器请求          │ 缓存命中，直接返回
     │                           │                          │
     
  epoll 同时盯: N 个 fd × M 种事件(EPOLLIN/EPOLLOUT)
  epoll_wait 醒来 → 遍历就绪的 → 根据 state 推进
```

## browser_fd vs target_fd

| | browser_fd | target_fd |
|------|-----------|----------|
| 怎么来的 | `accept()` 被动接收 | `socket()`+`connect()` 主动创建 |
| 底层类型 | int (socket) | int (socket) |
| 代理先做什么 | 读它（收请求） | 写它（发请求） |
| 代理后做什么 | 写它（回传响应） | 读它（收响应） |
| 缓存命中时 | 只回写 | 不创建 |

**底层完全相同，区别只在状态机如何使用。**

## epoll 底层原理

```
      网卡收到数据
           │
    硬件中断 → CPU
           │
    网卡驱动 → 以太网层 → IP层 → TCP层
           │
    TCP 找到对应 socket
    数据写入 socket 接收缓冲区
           │
    检查: 这个 socket 注册到 epoll 了吗?
    有 → 加到 epoll 就绪链表 → 唤醒等待进程
           │
    epoll_wait() 返回
    你的程序被唤醒，遍历就绪的 fd
```

**epoll = 在内核维护一棵红黑树（所有注册fd）+ 一条就绪链表（有数据的fd）。把"轮询100个fd"变成"只处理有数据的那个"。**
