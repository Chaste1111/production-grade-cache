# 计算机网络答辩 — 老师可能问的问题及参考答案

> 基于 HTTP 代理缓存服务器项目的深度准备

---

## 一、TCP 协议相关

### Q1: TCP 三次握手的过程？为什么是三次不是两次？

**答：**

```
客户端                        服务器
  SYN, seq=x       ──────────→
  ←──────────  SYN+ACK, seq=y, ack=x+1
  ACK, seq=x+1, ack=y+1 ──→
```

两次握手的问题：如果客户端第一个 SYN 因为网络延迟很久才到服务器，服务器回复 SYN+ACK 后以为连接建立了，但客户端早就放弃了。三次握手让客户端最后确认一次——"我确实要建立连接"。防止历史连接被误建立。

### Q2: 四次挥手为什么是四次？TIME_WAIT 有什么用？

**答：** 因为 TCP 是全双工的——每个方向独立关闭。一方说"我说完了"（FIN），对方确认"知道了"（ACK），但对方可能还有数据要发，等对方也说"我说完了"（FIN），再确认。所以需要四次。

TIME_WAIT 等 2MSL（约4分钟）：
- 确保最后的 ACK 对方收到了（如果对方没收到会重发 FIN）
- 让旧连接的所有数据包在网络中消失，防止干扰新连接

### Q3: 你的代理里 TCP 连接怎么管理的？有几个 TCP 连接？

**答：** 一次普通 HTTP 代理请求有**两次独立的 TCP 连接**：

```
浏览器 ←→ 代理（连接1）  +  代理 ←→ 目标服务器（连接2）
```

每个连接三次握手、数据传输、四次挥手都是独立的。缓存命中时只建立连接1，省略了连接2。

### Q4: 非阻塞 connect 怎么知道连接建立成功了？

**答：** `connect()` 在非阻塞 socket 上立即返回 `-1`，errno 为 `EINPROGRESS`（不是错误）。然后把目标 fd 注册到 epoll 监听 `EPOLLOUT` 事件。TCP 三次握手完成后，socket 变为可写，epoll 通知 `EPOLLOUT` 就绪——此时连接建立成功。

## 二、HTTP 协议相关

### Q5: HTTP 请求报文的结构是什么？请求行长什么样？

**答：**

```
请求行：  GET http://www.baidu.com/ HTTP/1.1\r\n
         ↑方法      ↑URL（正向代理是完整URL） ↑协议版本

请求头：  Host: www.baidu.com\r\n
         User-Agent: curl/8.18.0\r\n
         Accept: */*\r\n

空行：    \r\n        ← 头体和正文的分隔符

请求体：  (GET 请求通常为空)
```

### Q6: 正向代理和反向代理在 HTTP 协议层面有什么区别？

**答：** 请求行不同。

```
正向代理：GET http://www.baidu.com/ HTTP/1.1   ← URL 是完整的
反向代理：GET / HTTP/1.1                       ← URL 只有路径
                     Host: www.baidu.com        ← 目标信息在 Host 头里
```

正向代理里浏览器知道自己配了代理，所以把完整 URL 发给代理。代理从 URL 中解析目标地址。反向代理对客户端透明。

### Q7: CONNECT 方法是什么？和 GET 有什么区别？

**答：** `CONNECT` 用来建立 HTTPS 隧道。它不是请求一个资源，而是请求代理建立一个 TCP 隧道到目标服务器。

```
CONNECT www.baidu.com:443 HTTP/1.1
```

代理收到后：
1. TCP 连接目标 443 端口
2. 返回 `HTTP/1.1 200 Connection Established`
3. 之后变成透明管道——浏览器和目标服务器之间的 TLS 握手和加密 HTTP 数据直接双向中继

代理在隧道中**完全看不懂传输内容**（加密的），只是搬运 TCP 字节流。

### Q8: 常见的 HTTP 状态码有哪些？你的代理返回了哪些？

**答：**

| 状态码 | 含义 | 代理何时返回 |
|--------|------|-------------|
| 200 OK | 成功 | 正常转发/缓存命中 |
| 200 Connection Established | 隧道就绪 | CONNECT 隧道建立 |
| 403 Forbidden | 禁止访问 | 黑名单拦截 |
| 502 Bad Gateway | 网关错误 | 目标服务器连不上 |

## 三、正向代理架构

### Q9: 浏览器怎么知道要把请求发给代理？

**答：** 浏览器里手动配置的。HTTP 代理地址填 `127.0.0.1:8888`。配了之后浏览器不再直接连接目标服务器，而是把所有请求（包括 GET、CONNECT 等）都发给代理，目标地址写在 HTTP 请求的 URL 里。

### Q10: 请求头修改模块做了什么？

**答：** 代理在转发前修改 HTTP 头——添加、删除、替换。比如添加 `X-Forwarded-For` 告诉目标"真实的客户端是谁"，添加 `X-Proxy-Server` 标识代理版本。服务器可以通过这些头做日志分析、地域判断等。

## 四、epoll 与非阻塞 IO

### Q11: epoll 和 select 的区别？为什么 epoll 更快？

**答：**

| | select | epoll |
|------|--------|-------|
| 数据结构 | 位图（fd_set），上限 1024 | 红黑树，无上限 |
| 如何找到就绪 fd | 遍历全部 fd O(n) | 直接从就绪链表取 O(1) |
| 每次调用 | 重新传入 fd 集合（用户态→内核态拷贝） | 只注册一次，内核记住 |
| 10000 连接时 | 每次遍历 10000 个 | 只处理就绪的几个 |

**select = 每次点名全班 100 人看谁举手。epoll = 举手的自动站到讲台前。**

### Q12: 为什么要设非阻塞？不设会怎样？

**答：** 如果 socket 是阻塞的，`read()` 没数据时线程会一直睡到数据来——整个进程无法做任何其他事。非阻塞下 `read()` 没数据时立即返回 `EAGAIN`，可以马上去处理其他连接。epoll 要求所有 fd 必须是非阻塞的，否则一个 fd 的阻塞操作会卡住整个事件循环。

### Q13: LT（水平触发）和 ET（边沿触发）的区别？

**答：**

- **LT（Level Triggered）**：缓冲区有数据就一直通知，直到读完。实现简单，不易丢事件。本项目使用默认的 LT 模式。
- **ET（Edge Triggered）**：只在缓冲区从空变为非空的那一瞬间通知一次。必须一次性读完所有数据（循环 read 到 EAGAIN），否则数据就丢了。Nginx 使用 ET 来减少系统调用次数。

### Q14: 连接状态机有几个状态？为什么需要状态机？

**答：** 7 个状态：READ_REQUEST → CHECKING → CONNECTING → FORWARD → READ_RESPONSE → WRITE_CLIENT → DONE（缓存命中时从 CHECKING 直接跳到 WRITE_CLIENT；CONNECT 隧道进入 TUNNEL 状态）。

非阻塞 IO 下一次 `read` 可能只读到半截 HTTP 请求，所以每个连接必须记住"上次处理到哪了"——当前在哪个状态、缓冲区攒了多少数据、写到了哪个偏移量。状态机就是把这些"记忆"结构化。

## 五、缓存与淘汰策略

### Q15: 为什么用 LRU 而不是 LFU 或 FIFO？

**答：**

| 策略 | 规则 | 翻车场景 |
|------|------|---------|
| LRU | 踢掉最久没访问的 | 一次性扫描污染缓存（可接受） |
| LFU | 踢掉访问次数最少的 | 历史热门现在不访问了，踢不掉 |
| FIFO | 踢掉最早进来的 | 不管是否是热门，乱踢 |

**LRU 基于"时间局部性"原理——刚被访问的短期内更可能再次被访问。这个假设在代理缓存场景下成立（用户反复访问同一个网站）。**

### Q16: 怎么实现 LRU 的 O(1) 操作？

**答：** 两个数据结构配合：

- **双向链表 (`std::list`)**：按访问时间排序，头部最新，尾部最旧。`splice` 移动节点只改指针 O(1)，`pop_back` 淘汰 O(1)
- **哈希表 (`std::unordered_map<string, list::iterator>`)**：URL → 链表节点位置，O(1) 查找

哈希表的 value 存的是链表迭代器（指针），两者指向同一个节点。`splice` 只动指针不动数据——缓存的 HTML 哪怕是 10MB，移动也是 O(1)。

### Q17: TTL 过期怎么实现的？

**答：** 每个缓存条目存一个 `expire_time` 时间戳。`get()` 时先检查——如果 `time(nullptr) > expire_time`，删掉这个条目并返回未命中。构造函数可配 TTL 秒数，0 表示永不过期。

### Q18: 和 Nginx 的缓存实现有什么异同？

**答：** 核心思路完全一样：一个数据结构负责查找（Nginx 用红黑树，我们用哈希表），一个数据结构负责排序（都用双向链表）。Nginx 用红黑树是因为缓存元数据存在共享内存——多个 worker 进程共用，红黑树不需要扩容，对共享内存友好。我们的哈希表在进程私有内存里，扩容不是问题。

## 六、安全相关

### Q19: 为什么黑名单在缓存之前检查？

**答：** 如果先查缓存后查名单，被禁网站之前缓存过的内容会命中返回——黑名单形同虚设。名单是第一道门，缓存是后面的抽屉。先过门，再翻抽屉。

### Q20: 127.0.0.1 和 192.168.x.x 为什么要屏蔽？

**答：** 防止**内网穿透**。攻击者通过代理访问 `http://127.0.0.1:3306/`，代理就去连自己的 MySQL 端口。内部服务通常没有外部认证，这意味着敏感数据可能泄露。同样，192.168.x.x 是内网地址段，外部攻击者可以通过代理摸到公司内网的数据库、文件服务器。

### Q21: HTTPS 隧道中代理能看到加密内容吗？

**答：** 不能。TLS 握手是浏览器和目标服务器之间直接进行的，代理只是建立了 TCP 隧道。浏览器用服务器的公钥加密数据，只有服务器能解密。代理在中间看到的全是密文，只是搬运工。

## 七、性能与优化

### Q22: 三阶段的性能差异证明了什么？

**答：** 10 个不同网站并发请求：

| 阶段 | 耗时 | 原因 |
|------|------|------|
| Stage1 阻塞 | ~25s | 串行，一次一个 |
| Stage3 epoll | ~3s | 并发，交替推进 |

**单次请求的速度没有变快——变快的是"不用排队等别人"。epoll 让 IO 等待时间重叠，CPU 利用率大幅提升。Nginx 一个 worker 扛几万并发就是靠这个。**

### Q23: 如果缓存满了，淘汰策略怎么选？

**答：** 当缓存条数达到容量上限，淘汰链表尾部的节点——即最久未被访问的（LRU）。淘汰时同时从哈希表中删除对应条目，链表和哈希表保持同步。

### Q24: 内存缓存和磁盘缓存的区别？什么时候用哪个？

**答：** 内存缓存（本项目）——快但容量小，适合热点数据。磁盘缓存（Nginx 做法）——慢但容量大，适合图片、视频等大文件。Nginx 的做法是内存存元数据（URL→文件路径，红黑树），磁盘存响应体（文件系统）。

## 八、设计与工程

### Q25: 为什么用接口（CacheInterface/LoggerInterface）而不是直接依赖具体类？

**答：** **依赖注入 + 面向接口编程。** `proxy_handler` 只依赖接口，不关心具体实现。以后换 TTL 缓存、磁盘缓存、远程日志——只改 `proxy_server.cpp` 里的一行 `new`，其他代码不动。这是设计模式中的策略模式。

### Q26: 项目中用了哪些 Linux 系统调用？各自什么作用？

**答：**

| 系统调用 | 作用 |
|---------|------|
| `socket()` | 创建 TCP socket |
| `bind()` | 绑定 IP 和端口 |
| `listen()` | 标记为被动 socket |
| `accept()` | 接受客户端连接 |
| `connect()` | 主动连接目标服务器 |
| `read()` / `write()` | 收发数据 |
| `close()` | 关闭连接 |
| `fcntl(F_SETFL, O_NONBLOCK)` | 设置非阻塞 |
| `epoll_create / epoll_ctl / epoll_wait` | epoll 事件驱动 |
| `gethostbyname()` | DNS 域名解析 |
| `setsockopt(SO_REUSEADDR)` | 端口复用 |

## 九、开放性问题

### Q27: 如果让你改进这个项目，你会加什么？

**答：**
1. **多线程 + epoll**：Nginx 的做法——多 worker 进程，每个独立 epoll，利用多核 CPU
2. **HTTPS 拦截（MITM）**：代理作为中间人签发证书，解密和检查 HTTPS 流量（企业安全场景）
3. **连接复用（Keep-Alive）**：同一个 TCP 连接处理多个 HTTP 请求，减少握手开销
4. **压缩传输**：gzip 压缩响应体，减少带宽
5. **限流和认证**：防止代理被滥用

### Q28: 这个项目让你最大的收获是什么？

**答：** 从 TCP socket 编程到 HTTP 协议解析，从阻塞 IO 到 epoll 事件驱动——把计算机网络课程的核心知识点全部实践了一遍。特别是看了 Nginx 源码后自己实现，真正理解了 Nginx 为什么高性能——不是魔法，是 epoll + 非阻塞 + 状态机的合理设计。

### Q29: 你对 HTTP/3 (QUIC) 有了解吗？

**答：** HTTP/3 基于 QUIC，底层是 UDP 而不是 TCP。把 TCP 的可靠传输、TLS 加密、多路复用都在 QUIC 层实现，避免了 TCP 的队头阻塞问题。不过代理 HTTP/3 需要完全不同的实现方式——UDP 没有"连接"的概念，代理逻辑跟 TCP 代理完全不同。这也是未来可以继续研究的方向。
